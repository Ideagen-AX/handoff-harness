## What a harness actually is

A language model on its own is just a text predictor. You give it text, it gives you text back. That's it. It can't open a file, send an email, remember what happened yesterday, or take a second step based on the result of a first step.

**A harness is all the scaffolding around the model that turns "text in, text out" into an agent that can actually do work.** It's the machinery that:

- **Feeds the model the right context** (instructions, relevant docs, the current state of things)
- **Gives it tools** and runs them when the model asks (read a file, call an API, send a message)
- **Loops** — takes the model's output, executes any actions, feeds the results back in, and repeats until the job is done
- **Keeps it on the rails** — permissions, guardrails, what it's allowed to touch, when to stop and ask a human

A useful analogy: the model is an engine. The harness is the rest of the car — the transmission, steering, dashboard, brakes, and the seatbelt. A brilliant engine sitting on the floor doesn't take you anywhere.

**You're already using a harness every day: Claude Code is one.** The model doesn't inherently know how to read your files or run bash — Claude Code (the harness) gives it those tools, manages what context the model sees, runs the tools when the model requests them, loops until your task is done, and asks your permission before risky actions. Your hackathon task is to build a purpose-built version of that same idea, aimed at design handoff.

## The anatomy of a harness (the 6 pieces you'll design)

1. **The loop** — the core cycle: model thinks → requests an action → harness runs it → result goes back to model → repeat until done. This is the beating heart.
2. **Instructions / system prompt** — the agent's job description, personality, rules, and definition of "done."
3. **Context management** — what information the agent gets and when. Too little and it's clueless; too much and it drowns. (For you: the design specs, the Figma data, prior handoff messages, etc.)
4. **Tools** — the concrete capabilities. Each is a function the model can invoke: read a Figma file, draft an email, post to a channel, look up a component spec.
5. **Guardrails & control** — permissions, approval gates, what it can't do, when to hand back to a human. Critical for anything that talks to the outside world (like sending comms).
6. **Evaluation** — how you know it's doing a good job. Test cases, examples of good vs. bad output, a way to measure it.

## The steps to build one

1. **Define the job precisely** — one clear task the agent owns end-to-end. ("Take a completed design and produce a developer handoff summary + notify the right people," for instance.)
2. **Map the workflow you do today** — the manual steps, the inputs you look at, the outputs you produce, the decisions you make. The agent is automating *this*.
3. **Inventory the tools it needs** — for each manual step, what capability does the agent need? (Read Figma? Access a spec doc? Draft a Teams message?)
4. **Write the instructions** — role, process, rules, output format, when to stop and ask you.
5. **Decide context strategy** — what you'll feed it up front vs. what it fetches via tools.
6. **Set guardrails** — especially: does it *send* communications, or *draft* them for your approval? (Big decision for a comms agent.)
7. **Build the loop** — wire model + tools + context together. Depending on your comfort, this might be as light as a well-structured Claude Code setup, or a small app using the Claude Agent SDK.
8. **Test on real examples** — run it against actual past handoffs, see where it breaks, refine.

The good news: you don't have to build the loop, permission system, or tool-execution machinery from scratch. Claude Code and the Claude Agent SDK give you those for free — your real design work is steps 1–6.

That matters because it collides with something about the assignment itself:

> Your team was asked to **build a harness**. If you drop in the Agent SDK, you're mostly *using* Claude Code's harness, not building one. The loop, tools, guardrails, and review gate come pre-assembled.

| | AI SDK (build the harness) | Agent SDK (import the harness) |

| **Vercel deploy** | Pure in-process, deploys clean | 150MB subprocess, needs warming/memory tuning |
| **Streaming to UI** | First-class, trivial | Works, but more plumbing |

## The one nuance that separates a good harness from a bloated one

The craft is **not** "give it everything." Two ways to deliver knowledge, and choosing well is the skill:

1. **Static reference** — curated docs the agent reads. Best for *stable, high-signal* knowledge: product overview, org map, SQA process, brand voice, DS tokens. (This is exactly the `specs/references/` slot we already have — right now it holds one placeholder.)
2. **Dynamic retrieval via tools** — the agent fetches what it needs at runtime. Best for *large or changing* knowledge: the codebase, live Jira tickets, the current DS. (Our `read_reference` and `fetch_prototype` tools are the seed of this.)

The governing principle is the **context pillar**: the *right* knowledge at the *right* time. Too much context is its own failure mode — it dilutes the model's attention, costs more, and slows things down. So we curate the stable stuff and fetch the rest on demand.

Really good question — and you've put your finger on the biggest conceptual gap in the current design. Let me be straight about it.

## The honest answer: right now it does *not* have the "before" state

It isn't diffing anything. When you feed it a prototype, the Understand stage has exactly three signals to work from:

1. **The prototype itself** — which is only the *after* state
2. **Your one-line note** — "what changed and why"
3. **Its background knowledge** — the product reference (what EHSQ-E is today) and the design-system reference (what components already exist)

From those, it *reconstructs* a plausible account of what changed. The `whatChanged` and `beforeAfter` fields in the brief are **inference, not comparison.** It's essentially reasoning: "here's the new state, here's a note saying it changed, and here's what I know already exists — therefore these are probably the changes."

## Why it hasn't obviously fallen over yet

Three things prop it up:
- The **design-system reference** gives it a real baseline *at the component level* — that's why `componentImpact` can say "Toolbar → extended, Data Table → used-as-is." It genuinely knows those already exist.
- Your **note** points it at the delta.
- The **human-review gate** is the safety net — you catch it if it misattributes something.

But the limitation is real and you should assume it: with no true baseline, it can **describe existing behavior as if it were new**, or **miss a change** that isn't visually obvious. It's reconstructing the "before" from memory and inference, and it will sometimes get that wrong.

## The design question underneath your question

There are actually two philosophies, and they lead to different architectures:

- **Detect the change** — give the agent the current state so it can genuinely diff old vs new. More rigorous; needs a baseline input.
- **Communicate a described change** — the *designer* already knows exactly what changed (you built it), so you tell the agent the delta and its job is to translate that into artifacts, not rediscover it. In real handoffs, this is what annotations are.

Neither is "correct" — the best handoff tools do both. Right now we're leaning almost entirely on the weakest version of the second (a single-line note).

## How we'd fix it — options to give it a real "before"

| Approach | What you'd provide | Trade-off |
|---|---|---|
| **Baseline URL (true diff)** | The *current* state too — the live app screen or the current design | Highest rigor; agent fetches both and compares. Small code change. |
| **Structured change note** | A short structured delta (added / changed / removed) instead of one line | Cheapest; leans on you as the authority on what changed |
| **Codebase / git diff** | Access to the product's current UI code | Truest for the *built* feature, but heavier — the codebase tool we deferred |

My recommendation: **baseline URL + a slightly richer note.** You give it the current-state URL, it diffs against your prototype, and the note fills gaps the visual diff can't see (intent, why). When no baseline is provided, it falls back to today's behavior — but I'd have it *label* those fields as "inferred, unverified" so the reviewer knows to check them.

### 🤖 Claude · 2026-07-08 21:38

Both — and that's not me dodging the question. It's the actual nature of the thing, and the distinction is worth getting precise about, because it's the crux of your whole assignment.

## The short version

A harness is *what turns a model into an agent*. So building the harness **is** how you build the agent — they're not two separate deliverables, they're the structure and the behavior of the same system. "Agent" describes what it *does* (pursues a goal, takes actions autonomously). "Harness" describes the machinery that *lets it* (the loop, tools, context, guardrails, I/O). You can't have one without the other.

## What's actually what, in our system

Here's the honest anatomy — because our system isn't one monolithic agent, it's a **harness that orchestrates one genuinely agentic component plus several plain model calls**:

| Part | Agent or harness? | Why |
|---|---|---|
| **Understand stage** (`ToolLoopAgent` + `fetch_prototype`, `read_reference`, `read_codebase`) | **This is the agent.** | A model in a loop, autonomously deciding which tools to call, exploring a codebase, choosing how to classify components, downgrading to "inferred" on its own judgment. That's agentic behavior. |
| **The 5 generators + dev handoff + component specs** | Neither — **plain model calls.** | Each is a single bounded `generateText` that transforms the brief into one artifact. No loop, no tools, no autonomous multi-step decisions. |
| **Everything else** — the pipeline control flow, the fan-out/offshoot branching, the tools, the specs/knowledge layer, the review UI, the human-approval guardrail, the streaming, the deploy | **This is the harness.** | Deterministic code that wires the model calls together, feeds them the right context, manages I/O, and keeps a human in the loop. |

So: **we built a harness that contains one small agent and drives a set of model calls.**

## The nuance that actually matters

We *deliberately* made most of it deterministic orchestration rather than one big free-roaming agent ("here's a prototype, go do the whole handoff"). That was a design choice, and for *this* use case it's the right one:

- **Reviewability** — you can see and trust each step; a comms tool for a regulated product shouldn't be a black box
- **Safety** — the human-approval gate is a hard guardrail, not a suggestion
- **Predictability & cost** — bounded calls beat an open-ended agent burning tokens deciding what to do

A single mega-agent would be more "impressive" in a naive sense but *less* controllable — worse for handoff comms where auditability and human judgment matter.

## For your hackathon

You're covered on both framings:
- The assignment asked you to **build a harness** — you did, and it's the bulk of the work (we picked the AI SDK specifically so you'd *build* it, not import Claude Code's).
- The event's goal is **an agent that does work on your behalf** — from where you sit, that's exactly the experience: point it at a prototype, it does the handoff. Under the hood, that "agent" is your harness orchestrating an agentic Understand step plus tailored generation.

Put simply: **the harness is what you built; the agent is what it runs.**
