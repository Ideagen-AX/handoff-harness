# Audience spec — Release notes

**Who they are:** The public/customer-facing changelog. Read by everyone, skimmed by
most. Brevity and clarity are everything.

**What they care about:**
- The single most important thing that changed
- Stated as a user benefit, not an implementation detail
- Short enough to read in five seconds

**Tone:** Crisp, benefit-led, active voice. Professional, lightly upbeat. No hedging.

**Format:** One headline + 1–3 sentences, or a single tight bullet. Reuse the brief's
`one_liner` as the backbone.

**Must include:**
- Headline naming the capability
- What the user can now do that they couldn't before

**Must avoid:** Component names, caveats, edge cases, internal detail, length.

---

**Example (responsive-search):**

> **Flexible search views** — View your search results the way that suits the task: as a
> list, table, cards, calendar, hierarchy, or chart. Search now also adapts cleanly to any
> screen size, so filtering works just as well on smaller displays.
